# Prakash Nandan Jha (PNJ) - Engineering Rules

## Repository Memory & Historical Context
Treat every connected Git repository as institutional knowledge.
Whenever GitHub access or local Git history is available:
Automatically inspect:
- Repository structure
- Commit history
- Branches
- Pull Requests
- README files
- Issues
- Discussions
- Tags
- Releases
- Architecture
- Folder organization
- Component patterns
- API patterns
- Database schema
- Design systems
- Motion systems
- AI implementations
- Authentication
- Security
- Testing
- Deployment
- Documentation

Build a mental model of how I engineer products. Do not simply inspect files; understand engineering decisions, evolution, coding style, recurring architecture, reusable components, and my preferences. Use this knowledge when making future engineering decisions. Never repeat mistakes already solved in previous repositories. Always prefer patterns that have proven successful.

## Engineering Memory
Whenever beginning a task, before writing code ask internally:
- Have I solved something similar before?
- Is there reusable code?
- Is there a better architecture from another repository?
- Can I improve consistency?
- Can I reduce duplication?
- Can I improve maintainability?

If yes, reuse those proven ideas. Do not reinvent existing solutions.

---

## Final Principle
Every repository should contribute to a growing understanding of my engineering philosophy.
Treat previous work as engineering knowledge, not as code to copy.
Continuously improve the current project using lessons learned from all previous repositories.
